SpeedySpec Testing Framework

Status

Library is still fairly new and is unstable. Development is focused on providing a testing framework for PHP that matches functionality in other languages. Web site will be updated as PHP development is completed.

Goals

  1. Provide cross language standard API.
  2. Separate library concerns into packages.
  3. Generative property-based testing.

Languages

The attempt is to provide a standard API for the below languages. This isn't to create a single standard per language. It is to create an API that is very much similar across all supported languages. So the generative and property-based testing will have similar names and work very similar across languages.

  • PHP (in-progress)
  • Python
  • JavaScript
  • Java
  • Swift
  • Go
  • Rust
  • Julia
  • C
  • C++
  • Ruby (maybe)

Packages

The purpose of splitting the framework into packages, is to allow the developer to replace parts of the library with other libraries. Every part should be independent of the other, so can be used with any existing library or framework.

Spec Test Library

The spec library sets up the test test groups and examples. Allowing the runner to execute the tests.

Test Runner Library

The test runner library provides options for output and implements a CLI to the framework.

Assertion Library

Standard assertion library based on xUnit, except in function form.

Should Assertion Library

Assertion library based on ShouldJS JavaScript library.

Expect Assertion Library

Assertion library based on ExpectJS JavaScript library.

Property-Based Testing Library

Generative functions for creating input data for test cases. Also includes providing an test function for testing final returned result.

Static Analysis Test Generation Library

Parse the code and generate test cases based on the above libraries. The goal isn't to cover every edge case or every case. The goal is to use static analysis to get the types, if possible and run the code in live environment testing the output and finding cases where code fails.

Kp>This library is really just an experiment on the viability of parsing code and generating spec tests based on existing code. The other part of the experiment is whether it is possible to generate as much code from existing test cases when the actual code does not exist. If it is possible to generate test cases from existing code, then how well can actual code be generated from just test cases? The initial passes probably won't work well, but I think it will be a success if it generates a skeleton and does as much code generation as possible. Algorithm process code won't be possible, but the goal is to reduce the amount of code that needs to be written overall.
